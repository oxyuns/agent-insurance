## 1. 프로토콜 개요

### 한 줄 정의

ERC-8183은 **Client → Provider → Evaluator** 3자 구조의 온체인 Job Escrow 상태 머신이다. 코어 컨트랙트는 에스크로와 상태 전환만 담당하고, 확장 기능은 훅(Hook) 컨트랙트로 위임한다.

### 설계 철학

| 원칙 | 구현 방식 |
|---|---|
| 최소주의 코어 | 코어는 상태 전환 + 자금 이동만. 비즈니스 로직은 훅에 |
| 업그레이드 가능성 | UUPS Proxy 패턴 |
| 재진입 방지 | `ReentrancyGuardTransient` (EIP-1153, Cancun EVM) |
| CEI 패턴 | 모든 상태변경 함수에서 Checks → Effects → Interactions 순서 보장 |
| 훅 격리 | 훅 화이트리스트 + ERC165 검증으로 악의적 훅 차단 |

### 역할 정의

```
Client    — 잡 생성자, 자금 공급자
Provider  — 작업 수행자, 결과물 제출자
Evaluator — 결과물 검증자, 최종 승인/거절 결정자
Admin     — 수수료·훅 화이트리스트 관리자 (ADMIN_ROLE)
```

## 3. Job Struct 완전 분석

```solidity
struct Job {
    uint256 id;           // 1부터 시작하는 순차 ID. 0이면 존재하지 않음
    address client;       // 잡 생성자 = msg.sender of createJob()
    address provider;     // 작업 수행자. createJob 시 address(0) 허용
    address evaluator;    // 결과 검증자. 반드시 지정 필요 (address(0) 불가)
    string description;   // 자유 형식 설명. 온체인 저장 → 길수록 가스 증가
    uint256 budget;       // 결제 금액. paymentToken의 decimals 단위
    uint256 expiredAt;    // Unix 타임스탬프 (초). 최소 block.timestamp + 5분
    JobStatus status;     // 현재 상태 (enum, 0~5)
    address hook;         // 훅 컨트랙트 주소. address(0) = 훅 없음
}
```

### 필드별 불변성 및 변경 가능성

| 필드 | 설정 시점 | 이후 변경 | 비고 |
|---|---|---|---|
| `id` | createJob | 불변 | 0이면 잡 미존재 판별에 사용 |
| `client` | createJob | 불변 | msg.sender 자동 설정 |
| `provider` | createJob 또는 setProvider | 한 번만 (address(0)→주소) | 주소 설정 후 변경 불가 |
| `evaluator` | createJob | 불변 | address(0) 허용 안 됨 |
| `description` | createJob | 불변 | |
| `budget` | setBudget (여러 번 가능) | fund() 이후 불변 | 마지막 setBudget 값이 유효 |
| `expiredAt` | createJob | 불변 | |
| `status` | 각 상태전환 함수 | 단방향 진행 | 되돌리기 불가 |
| `hook` | createJob | 불변 | |

### JobStatus enum 값

```solidity
enum JobStatus {
    Open,       // 0 — 잡 생성됨
    Funded,     // 1 — 자금 에스크로됨
    Submitted,  // 2 — 결과물 제출됨
    Completed,  // 3 — 완료 (provider 지급)
    Rejected,   // 4 — 거절 (client 환불)
    Expired     // 5 — 만료 (client 환불)
}
```

## 4. 상태 머신 (State Machine)

### 전체 전환 다이어그램

```
                   [createJob]
                       │
                    ┌──▼──┐
              ┌─────┤ Open ├──────────────────────────┐
              │     └──┬──┘                           │
              │        │ setBudget() [provider]        │ reject() [client only]
              │        │ setProvider() [client]        │
              │        │ fund() [client]               │
              │     ┌──▼─────┐                        │
              │     │ Funded │                         │
              │     └──┬─────┘                        │
              │        │ submit() [provider]           │ reject() [evaluator]
              │     ┌──▼─────────┐                    │ claimRefund() [after expiry]
              │     │ Submitted  │                     │
              │     └──┬─────────┘                    │
              │        │ complete() [evaluator]        │ reject() [evaluator]
              │        │                              │ claimRefund() [after expiry]
           ┌──▼──┐  ┌──▼──────────┐  ┌──────────┐  ┌──▼──────┐
           │Rej. │  │ Completed   │  │ Expired  │  │Rejected │
           │(Open│  │             │  │          │  │(Fund/Sub│
           └─────┘  └─────────────┘  └──────────┘  └─────────┘
```

### 특수 경로: 0예산 직행 제출

`Open` 상태에서 `budget == 0`이고 `setBudget()`이 호출된 적 있으면 `fund()` 없이 바로 `submit()` 가능.

```solidity
// submit() 내부 조건 (코드 원문)
if (
    job.status != JobStatus.Funded &&
    (job.status != JobStatus.Open || job.budget > 0)
) revert WrongStatus();
// 즉: Funded가 아니고, Open이지만 budget > 0이면 revert
// → Open + budget == 0이면 통과
```

## 5. 함수별 완전 명세

### 5-1. `createJob()`

```solidity
function createJob(
    address provider,    // address(0) 허용 (나중에 setProvider로 설정)
    address evaluator,   // address(0) 불가 → ZeroAddress revert
    uint256 expiredAt,   // block.timestamp + 5분 초과여야 함
    string calldata description,
    address hook         // whitelistedHooks에 있어야 함. address(0)은 화이트리스트에 초기 등록됨
) external nonReentrant returns (uint256)
```

**동작 순서:**
1. 유효성 검사 (evaluator, expiredAt, hook 화이트리스트, ERC165)
2. jobCounter++ 후 Job 생성
3. `JobCreated` 이벤트 발행
4. `_afterHook(createJob.selector, abi.encode(client, provider, evaluator))` — **after only**

**주의:** `createJob`은 `beforeAction` 없이 `afterAction`만 호출된다.

---

### 5-2. `setProvider()`

```solidity
function setProvider(uint256 jobId, address provider_) external
// nonReentrant 없음
// 훅 없음
```

**조건:**
- `job.status == Open`
- `msg.sender == job.client`
- `job.provider == address(0)` (미설정 상태여야 함)
- `provider_ != address(0)`

**주의:** 이 함수에는 `nonReentrant`가 없다. 단독으로는 재진입 위험이 없지만 외부 컨트랙트에서 연쇄 호출 시 유의.

---

### 5-3. `setBudget()`

```solidity
function setBudget(
    uint256 jobId,
    uint256 amount,      // 0도 유효
    bytes calldata optParams
) external nonReentrant
```

**조건:**
- `job.status == Open`
- `msg.sender == job.provider`

**중요 동작:**
- `amount == 0`도 정상 처리됨
- `jobHasBudget[jobId] = true` 로 "명시적 setBudget 호출" 기록
- **여러 번 호출 가능 — 마지막 호출이 실효 예산**
- 훅: `_beforeHook` → 예산 설정 → `_afterHook`

**data encoding (훅으로 전달):**
```solidity
abi.encode(msg.sender, amount, optParams)
// (address caller, uint256 amount, bytes optParams)
```

---

### 5-4. `fund()`

```solidity
function fund(
    uint256 jobId,
    bytes calldata optParams
) external nonReentrant
```

**조건:**
- `job.status == Open`
- `msg.sender == job.client`
- `job.provider != address(0)`
- `block.timestamp < job.expiredAt`

**중요 동작:**
- `job.status = Funded` (상태 먼저 변경 — CEI 패턴)
- `job.budget > 0`인 경우에만 `safeTransferFrom` 실행
- `budget == 0`이면 토큰 이동 없이 상태만 전환됨

**data encoding:**
```solidity
abi.encode(msg.sender, optParams)
// (address caller, bytes optParams)
```

---

### 5-5. `submit()`

```solidity
function submit(
    uint256 jobId,
    bytes32 deliverable,  // 결과물 해시 (keccak256 등)
    bytes calldata optParams
) external nonReentrant
```

**조건 (복잡):**
```
허용:
  - job.status == Funded
  - job.status == Open && job.budget == 0  ← 특수 경로

불허:
  - job.status == Open && job.budget > 0   ← WrongStatus
  - 그 외 모든 상태                          ← WrongStatus
```

**data encoding:**
```solidity
abi.encode(msg.sender, deliverable, optParams)
// (address caller, bytes32 deliverable, bytes optParams)
```

---

### 5-6. `complete()`

```solidity
function complete(
    uint256 jobId,
    bytes32 reason,
    bytes calldata optParams
) external nonReentrant
```

**조건:**
- `job.status == Submitted`
- `msg.sender == job.evaluator`

**수수료 분배 로직:**
```solidity
uint256 amount = job.budget;
uint256 platformFee = (amount * platformFeeBP) / 10000;  // 정수 나눗셈 (버림)
uint256 evalFee    = (amount * evaluatorFeeBP) / 10000;  // 정수 나눗셈 (버림)
uint256 net        = amount - platformFee - evalFee;

if (platformFee > 0) paymentToken.safeTransfer(platformTreasury, platformFee);
if (evalFee > 0)     paymentToken.safeTransfer(job.evaluator, evalFee);
if (net > 0)         paymentToken.safeTransfer(job.provider, net);
```

**수수료가 0이면 transfer 자체를 건너뜀** (가스 절약).

**data encoding:**
```solidity
abi.encode(msg.sender, reason, optParams)
// (address caller, bytes32 reason, bytes optParams)
```

---

### 5-7. `reject()`

```solidity
function reject(
    uint256 jobId,
    bytes32 reason,
    bytes calldata optParams
) external nonReentrant
```

**호출 권한 (상태에 따라 다름):**

| 현재 상태 | 호출 가능자 | 환불 |
|---|---|---|
| Open | client only | 없음 (budget 미에스크로) |
| Funded | evaluator only | budget 전액 client에게 |
| Submitted | evaluator only | budget 전액 client에게 |
| 기타 | — | WrongStatus revert |

**환불 조건:**
```solidity
if (
    (prev == JobStatus.Funded || prev == JobStatus.Submitted) &&
    job.budget > 0
) {
    paymentToken.safeTransfer(job.client, job.budget);
}
```

**data encoding:**
```solidity
abi.encode(msg.sender, reason, optParams)
// (address caller, bytes32 reason, bytes optParams)
```

### 5-8. `claimRefund()`

```solidity
function claimRefund(uint256 jobId) external nonReentrant
// 훅 없음 — 의도적 설계 결정
```

**조건:**
- `job.status == Funded` 또는 `Submitted`
- `block.timestamp >= job.expiredAt`

**호출 권한:** 누구나 (client가 아니어도 됨). 자금은 항상 `job.client`에게 전송됨.

**훅이 없는 이유:** 악의적 훅이 `beforeAction`에서 revert해 환불을 영구 차단하는 시나리오를 원천 방지.

## 6. 훅 시스템 (Hook System)

### IACPHook 인터페이스

```solidity
interface IACPHook is IERC165 {
    function beforeAction(uint256 jobId, bytes4 selector, bytes calldata data) external;
    function afterAction(uint256 jobId, bytes4 selector, bytes calldata data) external;
}
```

### 훅 호출 타이밍

| 함수 | beforeAction | afterAction |
|---|---|---|
| createJob | 없음 | 있음 |
| setBudget | 있음 | 있음 |
| fund | 있음 | 있음 |
| submit | 있음 | 있음 |
| complete | 있음 | 있음 |
| reject | 있음 | 있음 |
| claimRefund | **없음** | **없음** |
| setProvider | **없음** | **없음** |

### selector별 data 인코딩 완전 레퍼런스

```solidity
// createJob — afterAction only
// selector = AgenticCommerce.createJob.selector
abi.encode(address client, address provider, address evaluator)

// setBudget
// selector = AgenticCommerce.setBudget.selector
abi.encode(address caller, uint256 amount, bytes optParams)

// fund
// selector = AgenticCommerce.fund.selector
abi.encode(address caller, bytes optParams)

// submit
// selector = AgenticCommerce.submit.selector
abi.encode(address caller, bytes32 deliverable, bytes optParams)

// complete
// selector = AgenticCommerce.complete.selector
abi.encode(address caller, bytes32 reason, bytes optParams)

// reject
// selector = AgenticCommerce.reject.selector
abi.encode(address caller, bytes32 reason, bytes optParams)
```

### 훅 구현 시 필수 패턴

```solidity
contract MyHook is IACPHook, ERC165 {
    address public immutable acp;

    // 1. ACP에서만 호출 가능하도록 반드시 체크
    modifier onlyACP() {
        require(msg.sender == acp, "Not ACP");
        _;
    }

    function beforeAction(uint256 jobId, bytes4 selector, bytes calldata data)
        external override onlyACP
    {
        // beforeAction에서 revert = 전체 트랜잭션 롤백
        // 의도적 차단에만 사용할 것
    }

    function afterAction(uint256 jobId, bytes4 selector, bytes calldata data)
        external override onlyACP
    {
        // afterAction에서 외부 컨트랙트 호출은 반드시 try/catch로 감쌀 것
        // 실패 시 complete()/reject() 전체가 롤백됨
        try externalContract.doSomething() {
            // 성공
        } catch {
            // 실패해도 메인 트랜잭션은 유지
        }
    }

    // 2. ERC165 지원 선언 필수 (없으면 createJob에서 revert)
    function supportsInterface(bytes4 interfaceId)
        public view override returns (bool)
    {
        return interfaceId == type(IACPHook).interfaceId
            || super.supportsInterface(interfaceId);
    }
}
```

### 훅 내부에서 ACP 재호출 금지

훅 내부에서 ACP 함수를 다시 호출하면 `ReentrancyGuardTransient`에 의해 revert된다. 훅은 상태 읽기(`getJob`)는 가능하지만 상태 변경 함수 호출은 불가.

```solidity
// 가능
IAgenticCommerce(acp).getJob(jobId); // view 함수는 OK

// 불가 — ReentrancyGuard에 막힘
IAgenticCommerce(acp).complete(jobId, reason, ""); // revert
```

### optParams 활용

각 함수의 마지막 인자 `bytes calldata optParams`는 훅으로 그대로 전달된다. 코어 컨트랙트는 이 값을 읽지 않는다. 커스텀 데이터를 훅에 전달하는 유일한 채널.

```solidity
// 예시: fund() 호출 시 보험 플랜 ID 전달
uint8 planId = 2;
bytes memory optParams = abi.encode(planId);
agenticCommerce.fund(jobId, optParams);

// 훅에서 파싱
function beforeAction(uint256 jobId, bytes4 selector, bytes calldata data) external {
    if (selector == IAgenticCommerce.fund.selector) {
        (address caller, bytes memory opt) = abi.decode(data, (address, bytes));
        uint8 planId = abi.decode(opt, (uint8));
    }
}
```

## 7. 수수료 메커니즘

### 수수료 계산

```
complete() 호출 시:

platformFee  = (budget × platformFeeBP) / 10000   → Solidity 정수 나눗셈 (버림)
evaluatorFee = (budget × evaluatorFeeBP) / 10000  → Solidity 정수 나눗셈 (버림)
net          = budget - platformFee - evaluatorFee → provider 수령

제약:
  platformFeeBP + evaluatorFeeBP <= 10000 (100%)
  각각 0 설정 가능
```

### 정수 나눗셈 버림 효과

작은 budget에서 수수료가 0으로 계산될 수 있다.

```
예시: budget = 99 (0.000099 USDC 가정)
     platformFeeBP = 200 (2%)

platformFee = (99 × 200) / 10000 = 19800 / 10000 = 1 (버림)
실제 비율 = 1/99 ≈ 1.01%  ← 설정값(2%)과 다름
```

매우 작은 budget의 잡에서는 수수료 비율이 의도와 다르게 적용됨. 시스템 설계 시 최소 budget 제약을 훅 레벨에서 강제하는 것을 권장.

### 수수료 설정 제약

`setPlatformFee`와 `setEvaluatorFee`는 서로 합계가 10000을 초과하지 않도록 상호 검사:

```solidity
function setPlatformFee(uint256 feeBP_, address treasury_) external {
    if (feeBP_ + evaluatorFeeBP > 10000) revert FeesTooHigh();
    // ...
}

function setEvaluatorFee(uint256 feeBP_) external {
    if (feeBP_ + platformFeeBP > 10000) revert FeesTooHigh();
    // ...
}
```

## 8. 보안 모델

### 8-1. UUPS 업그레이드 패턴

```
사용자 → Proxy (ERC1967) → Implementation (AgenticCommerce)
                                    ↓
                           _authorizeUpgrade()
                           onlyRole(DEFAULT_ADMIN_ROLE)
```

- 업그레이드 권한은 `DEFAULT_ADMIN_ROLE`만 보유
- `ADMIN_ROLE`과 `DEFAULT_ADMIN_ROLE`은 별도 관리 가능 (더 세밀한 권한 분리)
- 초기화 시 `deployer`가 두 역할 모두 취득

### 8-2. ReentrancyGuardTransient

EIP-1153 (Cancun 하드포크)의 `transient storage`를 사용한 재진입 방지:

```
기존 ReentrancyGuard:
  진입 시 storage slot 1로 설정 (SSTORE = 20,000 gas)
  종료 시 storage slot 0으로 초기화 (SSTORE = 2,900 gas)
  총 ~22,900 gas

ReentrancyGuardTransient:
  진입 시 transient slot 1로 설정 (TSTORE = 100 gas)
  트랜잭션 종료 시 자동 초기화 (추가 비용 없음)
  총 ~100 gas
```

하드포크 요구사항: `hardhat.config.js`에 `evmVersion: "cancun"` 명시 필요.

### 8-3. CEI 패턴 (Checks-Effects-Interactions)

모든 상태변경 함수가 다음 순서를 준수:

```
1. Checks    — 모든 유효성 검사
2. Effects   — 상태 변수 변경
3. Hook      — _beforeHook (External call — 재진입 방지 필요)
4. Interactions — 토큰 이동 (External call)
5. Hook      — _afterHook (External call)
```

`fund()` 예시:
```solidity
function fund(uint256 jobId, bytes calldata optParams) external nonReentrant {
    Job storage job = jobs[jobId];

    // 1. Checks
    if (job.id == 0) revert InvalidJob();
    if (job.status != JobStatus.Open) revert WrongStatus();
    if (msg.sender != job.client) revert Unauthorized();
    if (job.provider == address(0)) revert ProviderNotSet();
    if (block.timestamp >= job.expiredAt) revert WrongStatus();

    bytes memory data = abi.encode(msg.sender, optParams);

    // 2+3. Effects + beforeHook
    _beforeHook(job.hook, jobId, msg.sig, data);   // 외부 호출이지만 상태변경 전 호출 가능 (상태는 아직 변경 전)

    // 3. Effects (상태 변경)
    job.status = JobStatus.Funded;

    // 4. Interactions (토큰 이동)
    if (job.budget > 0) {
        paymentToken.safeTransferFrom(job.client, address(this), job.budget);
    }

    emit JobFunded(jobId, job.client, job.budget);

    // 5. afterHook
    _afterHook(job.hook, jobId, msg.sig, data);
}
```

### 8-4. 훅 화이트리스트 + ERC165 이중 검증

```solidity
// createJob 내부
if (!whitelistedHooks[hook]) revert HookNotWhitelisted();
if (hook != address(0)) {
    if (!ERC165Checker.supportsInterface(hook, type(IACPHook).interfaceId))
        revert InvalidJob();
}
```

- `address(0)`은 초기화 시 `whitelistedHooks[address(0)] = true`로 미리 등록됨
- `address(0)` 외 모든 훅은 Admin이 명시적으로 화이트리스트에 추가해야 함
- ERC165로 실제 인터페이스 지원 여부를 런타임 검증

### 8-5. 훅의 공격 벡터와 방어

| 공격 | 방어 |
|---|---|
| 악의적 훅이 funds 탈취 | 훅은 ACP의 상태 변경 함수 호출 불가 (ReentrancyGuard) |
| 악의적 훅이 환불 차단 | claimRefund에 훅 없음 |
| 악의적 훅이 상태 변경 | 훅은 ACP 상태에 직접 접근 불가 |
| beforeAction revert로 서비스 거부 | 관리자가 화이트리스트에서 제거 가능 |

## 9. 숨겨진 엣지 케이스

### 9-1. 0예산 잡 (Free Job)

```
시나리오: AI 에이전트가 신뢰 관계에서 무료로 작업 수행

createJob(provider, evaluator, expiry, desc, hook)
setBudget(jobId, 0, optParams)  ← 0 예산 명시
// fund() 호출 없이 바로:
submit(jobId, deliverable, optParams)  ← Open 상태에서 직접 제출 가능
complete(jobId, reason, optParams)      ← 토큰 이동 없이 완료
```

`submit()` 조건 코드:
```solidity
if (
    job.status != JobStatus.Funded &&
    (job.status != JobStatus.Open || job.budget > 0)
) revert WrongStatus();
```

- `jobHasBudget[jobId]`가 true (setBudget 호출됨) + `job.budget == 0` → 직행 제출 가능
- setBudget 없이 budget이 기본값 0인 상태로는 이 경로 불가 (jobHasBudget이 false)

### 9-2. setBudget 재호출로 예산 협상

```
시나리오: provider가 예산을 여러 번 변경하며 협상

setBudget(jobId, 100e6, ...)  ← 100 USDC 제안
setBudget(jobId, 80e6, ...)   ← 80 USDC로 재협상
setBudget(jobId, 90e6, ...)   ← 90 USDC 합의

fund() 시점의 budget = 90e6  ← 마지막 값만 유효
```

훅이 있다면 setBudget 호출 이력을 afterAction에서 오프체인에 기록하고 협상 과정을 추적할 수 있음.

### 9-3. provider가 fund() 전에 reject() 불가

Open 상태의 reject()는 client만 가능. provider는 Open 상태에서 잡을 취소할 방법이 없음. 설계 의도: provider가 수락한 잡에서 이탈을 막기 위함. 이를 우회하려면 훅에서 provider에게 취소 권한을 부여하는 로직을 구현해야 함.

### 9-4. expiredAt 체크 시점

`fund()`만 `block.timestamp < job.expiredAt`를 검사함. `submit()`과 `complete()`는 만료 시간을 체크하지 않음. 즉, 만료된 잡에서도 (claimRefund 전까지) submit과 complete가 가능함. claimRefund와 complete가 경쟁 조건에 놓일 수 있음.

```
시나리오: 만료 직후 경쟁 조건
1. 잡 만료 (block.timestamp >= expiredAt)
2. provider가 submit() 호출 → 성공
3. evaluator가 complete() 호출 → 성공 (provider 지급)
   OR client가 claimRefund() 호출 → 성공 (client 환불)
   → 먼저 실행된 트랜잭션이 승리
```

### 9-5. paymentToken 불변성

`paymentToken`은 초기화 시 설정되고 이후 변경 방법 없음. 멀티토큰 결제를 지원하려면 별도 래퍼 컨트랙트가 필요.

### 9-6. description 가스 비용

`string description`은 온체인에 저장됨. 길이에 따라 가스 비용이 선형으로 증가.

```
빈 문자열:      ~21,000 gas (기본)
100자 문자열:   ~25,000 gas
1000자 문자열:  ~50,000+ gas
```

프로덕션에서는 description을 IPFS 해시 또는 짧은 ID로 대체하는 것을 권장.

## 11. 온체인 데이터 레이아웃

### Storage Slot 패킹

Solidity는 32바이트 단위로 슬롯을 할당하고, 같은 슬롯에 여러 작은 변수를 패킹.

```
Job struct storage layout (순서대로):
  Slot 0: id (uint256, 32bytes) — 단독 슬롯
  Slot 1: client (address, 20bytes)
  Slot 2: provider (address, 20bytes)
  Slot 3: evaluator (address, 20bytes)
  Slot 4: description (string — 동적, 별도 슬롯)
  Slot 5: budget (uint256, 32bytes)
  Slot 6: expiredAt (uint256, 32bytes)
  Slot 7: status (JobStatus enum = uint8, 1byte)
           hook (address, 20bytes)
           → 두 변수가 같은 슬롯에 패킹 (21bytes 합산, 32bytes 슬롯 내 여유)
```

### getJob() 반환값 읽기 (JavaScript/TypeScript)

## 12. 통합 가이드 (Integration Guide)

### 12-1. 기본 E2E 플로우 (훅 없음)

```typescript
import { ethers, upgrades } from "hardhat";

// 1. 배포
const Core = await ethers.getContractFactory("AgenticCommerce");
const core = await upgrades.deployProxy(Core, [usdcAddress, treasuryAddress], { kind: 'uups' });

// 2. 잡 생성
const expiry = Math.floor(Date.now() / 1000) + 3600; // 1시간 후
const tx = await core.connect(client).createJob(
    providerAddress,
    evaluatorAddress,
    expiry,
    "작업 설명",
    ethers.ZeroAddress  // 훅 없음
);
const receipt = await tx.wait();
const event = receipt.logs.find(l => l.fragment?.name === 'JobCreated');
const jobId = event.args.jobId;

// 3. 예산 설정 (provider)
await core.connect(provider).setBudget(jobId, ethers.parseUnits("20", 6), "0x");

// 4. 자금 공급 (client)
await usdc.connect(client).approve(await core.getAddress(), ethers.parseUnits("20", 6));
await core.connect(client).fund(jobId, "0x");

// 5. 결과물 제출 (provider)
const deliverable = ethers.keccak256(ethers.toUtf8Bytes("https://result.url"));
await core.connect(provider).submit(jobId, deliverable, "0x");

// 6. 완료 승인 (evaluator)
const reason = ethers.encodeBytes32String("approved");
await core.connect(evaluator).complete(jobId, reason, "0x");
```

### 12-2. 훅 연동 플로우

### 12-3. 이벤트 인덱싱 (오프체인 상태 추적)

```typescript
// The Graph 또는 직접 이벤트 구독

// 잡 생성 감지
core.on("JobCreated", (jobId, client, provider, evaluator, expiredAt, hook) => {
    db.insertJob({ jobId, client, provider, evaluator, expiredAt, hook, status: 'Open' });
});

// 상태 전환 감지
core.on("JobFunded", (jobId, client, amount) => {
    db.updateJobStatus(jobId, 'Funded');
});

core.on("JobCompleted", (jobId, evaluator, reason) => {
    db.updateJobStatus(jobId, 'Completed');
});

core.on("JobRejected", (jobId, rejector, reason) => {
    db.updateJobStatus(jobId, 'Rejected');
});

// 환불 추적
core.on("Refunded", (jobId, client, amount) => {
    console.log(`Job ${jobId}: ${amount} refunded to ${client}`);
});
```

### 12-4. 자주 하는 실수

```
1. provider approve 없이 setBudget 호출
   → setBudget 자체는 토큰 이동 없음. fund()에서 safeTransferFrom 발생.
   → client가 fund() 전에 ACP 주소로 approve 필요.

2. 훅에서 ACP 재호출
   → ReentrancyGuard에 막혀 revert.
   → 훅 내부에서는 getJob() 등 view 함수만 호출 가능.

3. ERC165 없는 훅 배포
   → createJob에서 InvalidJob revert.
   → 반드시 supportsInterface 구현 필요.

4. afterAction에서 외부 호출 시 try/catch 생략
   → 외부 호출 실패 시 complete()/reject() 전체 롤백.
   → 외부 호출은 항상 try/catch로 감쌀 것.

5. claimRefund에 훅 기반 로직 의존
   → claimRefund는 훅을 호출하지 않음.
   → 만료 처리 로직을 훅에 넣으면 실행되지 않음.
```
